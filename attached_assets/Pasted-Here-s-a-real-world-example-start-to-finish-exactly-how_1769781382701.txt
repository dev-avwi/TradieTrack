Yeah — it can be secure enough for your use case if you do it properly, and you don’t need bank-level security. But invoice# + phone by itself is only “okay” if you add a couple safeguards, because it’s basically “something you know” + “something you know”. The good news: adding a verification code (OTP) is not too hard and it upgrades trust a lot.

Here’s the clean recommendation.

What’s “secure enough” for TradieTrack portals?
Your portal is exposing:

invoice amount + line items

maybe address

photos (potentially sensitive)

payment link

So you want protection against:

random guessing

someone who knows the customer’s phone

spam scripts hammering the portal

You don’t need perfect security, but you do need “hard to abuse”.

Best approach (simple + strong)
✅ Use Magic links as the primary access

This is already secure enough in practice because tokens are long and unguessable.

Example:
/portal/i/3F7B... (really should be a long token like 32+ chars)

That’s your main path.

✅ For “lost link” recovery: use Phone OTP (verification code)

Instead of invoice# + phone.

Flow:

Client enters phone number on /portal

System sends a 6-digit code by SMS

Client enters code

System shows their invoices/jobs (or just invoices) and they pick

This is way more secure because it proves they actually control the phone.

Is this too hard?

No — because you already want Twilio for tradie SMS anyway. OTP is basically:

generate code

store hash + expiry

send SMS

verify

That’s very doable.

What I would ship (polished and realistic)
Portal access methods (final)

SMS magic link (default, easiest, most secure)

Lost link? Verify by SMS code (OTP recovery)

That’s it. No invoice number needed.

If you still want invoice numbers

You can still show invoice numbers as convenience, but don’t rely on them as “security”.

If you insist on invoice lookup, make it safer like this:

✅ Invoice number should be non-guessable: INV-1042-K7Q (random suffix)
✅ Add rate limiting + lockout
✅ Don’t reveal whether invoice exists until after verification
✅ Only allow a few tries per IP/phone per hour

But honestly — OTP is cleaner and more modern.

Recommended security minimums (keep it simple)
For magic links

Token = long, random (at least 128-bit)

Expire tokens after X time for quotes (e.g., 30–90 days)

For invoices/receipts you can keep longer, but allow “regenerate link”

Optional: if link is opened on a totally new device, require OTP (later)

For OTP

6 digits

Expire in 10 minutes

5 attempts max

Rate limit sends (e.g., 3 per hour per phone)

Store hashed OTP, not plain text

General

Don’t show client photos on invoice recovery unless verified

Log access events (basic)

Straight answer: invoice# + phone vs verification code
Invoice# + phone

✅ easiest to build
⚠️ weaker security
⚠️ people mistype invoice numbers
⚠️ invoice numbers can leak on PDFs

Verification code (OTP)

✅ still easy to build
✅ much better security
✅ better UX (“send me a code”)
✅ fits your SMS ecosystem perfectly

My pick: OTP recovery, no invoice number required.
