========================================================
A) REQUIRED ACCOUNT MODEL + ROLE LOGIC
========================================================

1) Define and enforce these concepts in code + database:
- Business/Workspace (one tradie business)
- Users (owner, admin/manager, worker)
- Membership (user belongs to business with a role)
- Subscription tier (FREE / PRO / TEAM)
- Subscription status (beta_free, trialing, active, past_due, canceled)
- Trial end timestamp (trial_ends_at)
- Seats: number of active worker memberships (for TEAM plan)

2) Role rules (must be enforced server-side, not just UI):
- Owner/admin can: billing, business settings, invite/remove members, see map/team locations (if enabled), export data
- Worker can: assigned jobs, job updates, time tracking, photos, job chat
- Worker cannot: billing, delete business, see all financials unless allowed by role permissions

3) Verify ALL checks are server-side:
- API endpoints must verify role before allowing actions
- Frontend gating alone is not enough

========================================================
B) PLAN TIERS — WHAT EACH PLAN CAN DO
========================================================

Implement clear plan gating with a single source of truth:
- A function like canUseFeature(business, featureKey) used across web+mobile+API

Required plan behavior:

- Core workflow works
- Basic quotes/invoices/PDF/email sending works
- Basic job tracking works
- BUT some advanced features can be limited (e.g. advanced automations, advanced branding, team size caps)

PRO
- Adds branding (logo/header/footer templates)
- Removes “Powered by TradieTrack”
- Automations (reminders/follow-ups)
- Additional templates / document customization
- (Optional) Accounting export (CSV)

TEAM
- Everything in PRO
- Multiple members (seats)
- Team map + locations
- Group chat + job chat for teams
- Dispatch/assignment board
- (Optional) advanced permissions + audit logs

IMPORTANT:
- If a feature is visible but not available in the current plan, show a clean upgrade modal.
- Do NOT show broken / unimplemented “Upgrade to unlock” flows.

C) BILLING MODES: BETA OFF → LIVE ON (CRITICAL)
========================================================

We need a “billing mode” switch that allows:

MODE 1: BETA (billing disabled)
- Everyone can use PRO/TEAM features for free (or controlled access)
- No Stripe charges or paywalls shown during Apple review if that risks rejection
- Any upgrade UI should be hidden or clearly marked “Beta: billing not enabled yet”

MODE 2: LIVE (billing enabled)
- New users start trial (e.g. 14 or 30 days)
- Collect card at trial start is OPTIONAL; implement it as a setting:
  - collect_card_on_trial_start = false for now
  - later can be true
- After trial ends: enforce paywall for gated features
- Existing beta users should be handled cleanly:
  - Either grandfather them with beta_free status until you migrate them
  - Or start their trial from the day billing is enabled

Implement:
- A single config flag (ENV or DB setting):
  BILLING_MODE = "beta" | "live"
- Ensure BOTH web + mobile respect this

Add an admin-only “Billing Mode” indicator in settings (not visible to customers).

========================================================

- TEAM seat price per worker (metered or quantity-based)

- Count active worker memberships in the business (excluding owner/admin if desired)

4) Webhooks:
- subscription created/updated/canceled
