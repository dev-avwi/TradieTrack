## 1) Account Types and What Each Can Do

### Roles (RBAC)

You need **Role-Based Access Control** with these roles:

**Owner (Company Owner)**

* Full access to company settings, billing, plans
* Create/edit users + roles
* View all jobs, all time entries, all invoices
* Approve timesheets
* Send client links (tracking portal)
* Manage templates (SMS/email, docs)
* Change branding (logo, name, contact, ABN)
* Audit logs

**Manager (Ops/Admin)**

* Create/edit jobs
* Assign workers/subcontractors
* View all team schedules + locations (if enabled)
* Approve/reject time entries (optional permission)
* Send client links
* Upload docs/photos
* Cannot change billing/plan/ownership
* Cannot delete company

**Worker (Employee)**

* See only assigned jobs
* Clock in/out, job notes, photos, receipts
* Share ETA updates (optional)
* Cannot view company finance unless allowed
* Cannot see other workers unless “team view” is enabled

**Subcontractor (External Worker)**

* Same as Worker, but:

  * Restricted to jobs they’re assigned
  * Limited access duration
  * Optional: can only upload docs + time, cannot edit job details

**Client (External Viewer)**

* No login required
* Views “JobRunner Link Portal”
* Sees only what you allow

✅ Key idea: **Permissions aren’t just roles** — you need a “permissions matrix” so an Owner can make a Manager limited or powerful.

## 2) Team Structure That Never Breaks

### Company → Teams → Users

* Company has many users
* Users belong to exactly 1 company (for now; multi-company later)
* Users have:

  * role
  * status (active, suspended)
  * verified phone/email
  * optional 2FA

### Job Assignment Model

A job can have:

* 1 assigned primary worker
* optional additional workers
* optional supervisor/manager

**Rule:** Workers only sync jobs they’re assigned to (privacy + performance).

---

## 3) “Perfect Sync” Rules (So Data Never Gets Messy)

### Offline-first rules

Workers often lose service, so enforce:

* Time entries are created locally first
* Photos/receipts upload when online
* GPS pings queue locally if offline
* Sync is “append-only” for logs (no overwriting history)
* Conflicts resolved with:

  * last-write-wins for editable fields
  * logs preserved for audit fields

### Audit Trail Required

Every important change must log:

* who did it
* when
* before vs after
* device id (optional)
* location (optional)

Track changes for:

* time edits
* job assignment changes
* invoice edits
* client portal link creation / resend
* photo deletions
* status changes

## 4) Team Operations Flows (How They Interact)

### A) Dispatch flow (Owner/Manager)

1. Create job
2. Add address + scope + scheduled window
3. Assign worker(s)
4. Attach any docs (plans PDFs etc.)
5. Click “Send Client Updates”
6. System generates **Client Portal Link**
7. Sends SMS + email with link

### B) Worker flow (On the day)

1. Worker opens job
2. Press “On my way”
3. App begins:

   * ETA tracking
   * location sharing to portal (if enabled)
4. Worker arrives → “Arrived”
5. Clock in (or auto prompt)
6. Upload before photos
7. Update notes/materials
8. Clock out
9. Upload after photos
10. Mark job complete

### C) Owner/Manager oversight

* Live map view of active jobs (optional)
* Sees worker statuses: On my way, Arrived, In progress, Done
* Can message worker inside app (ChatHub)
* Can message client via portal updates

## 6) Security and Privacy Requirements (Non-negotiable)

### Link Security

* Portal link contains a long random token
* Token expires (default 7 days after job completion)
* Token can be revoked instantly by Owner/Manager
* Token is read-only unless client action is allowed
* Optional PIN view for sensitive jobs

### Location Sharing Safety

* Location sharing only during:

  * “On my way” → “Arrived”
* Stops automatically at arrival OR after max time (e.g., 2 hours)
* Client never sees full worker history—only job route
* Worker can pause sharing (with audit log)

### Photo Safety

* Before/after photos only visible to that job’s link
* Deleting photos requires Owner/Manager permission OR creates an audit log
