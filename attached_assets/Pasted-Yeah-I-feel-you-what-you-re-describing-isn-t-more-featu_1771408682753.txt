1) Uber-Style Tracking: make it “trustworthy”, not just “cool”

The 4 states that must be real

You need a clear worker status machine:
	•	OFFLINE / OFFSHIFT
	•	AVAILABLE
	•	EN_ROUTE
	•	ON_SITE
	•	WORKING (timer running)

Rules
	•	EN_ROUTE requires: job selected + “navigate” action OR explicit “Start travel”
	•	ON_SITE triggers via: GPS geofence OR “Arrived” tap (manual fallback)
	•	WORKING triggers when timer starts
	•	OFFLINE triggers when no pings in X mins + app backgrounded

Ping strategy (battery-safe)
	•	Only high frequency when EN_ROUTE or ON_SITE
	•	Low frequency when AVAILABLE
	•	No pings when OFFSHIFT

Reliability features (this is what makes it business-ready)
	•	Last seen timestamp on every worker
	•	Signal quality indicator (good/weak/offline)
	•	Fallback: “Worker can’t share live location → show last known”
	•	Audit: store location events as “breadcrumbs” during travel only (privacy + storage)

⸻

2) Notifications: build a single “Routing Engine” (so it never misses)

Right now most apps fail because notifications are scattered (some in SMS, some in push, some in email, some not at all).

Build one notification pipeline

Every event becomes a notification record:

Events (examples)
	•	Job assigned / unassigned
	•	Job rescheduled
	•	Worker marked EN_ROUTE / ON_SITE / DONE
	•	Quote viewed / accepted / declined
	•	Invoice sent / paid / overdue
	•	Subcontractor payable submitted / approved
	•	Timer started and left running too long
	•	New client SMS / portal message

Delivery channels
	•	Push (primary for workers)
	•	In-app (audit + history)
	•	Email (owners/managers summary)
	•	SMS (only where essential)

Recipient rules (this is your missing “glue”)

For each event, define exactly who gets it:

Example: “Job assigned”
	•	Assigned worker ✅
	•	Owner ✅
	•	Job’s manager ✅ (if your app has “job manager” role)
	•	Office admin ✅ (optional toggle)
	•	NOT the whole team

Example: “Worker arrived on-site”
	•	Owner ✅
	•	Job manager ✅
	•	Client ✅ (optional: “Your tradie has arrived”)

This is a rules table Replit can implement once and reuse forever.

3) Managers + Team Improvements: add a proper “Job Ownership Model”

Most team systems feel bad because jobs aren’t owned clearly.

Add 2 new concepts:

A) Job Manager (internal)

Every job has:
	•	jobManagerUserId (owner or admin or supervisor)

This person:
	•	approves changes
	•	gets escalations
	•	is accountable

B) Worker Role Tiers
	•	Worker (field)
	•	Supervisor (can reassign + approve timesheets)
	•	Admin (office)

This gives real structure.

4) Team features that need upgrading (the “business-grade” list)

If JobRunner is going to feel like real ops software, team needs:

A) Assignment confirmation
	•	Worker gets assignment → must Accept / Acknowledge (optional)
	•	If not acknowledged in 10 mins → escalate to manager

B) Shift/availability
	•	Worker availability status
	•	Time-off requests
	•	“Do not assign me outside X hours” (simple)

C) Escalations

If something happens, it must escalate:
	•	no location updates for 15 mins while EN_ROUTE → notify manager
	•	timer running past shift end → notify worker + manager
	•	job overdue with no activity → notify owner/manager

D) Team activity feed that matters

Not noise.
Only operational changes.
