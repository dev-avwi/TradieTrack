# JobRunner — Claude Agent Context

> **Read this in full before suggesting any change.** Most generic Swift/iOS advice will break this project because it's Expo-managed, not a hand-rolled native app.

---

## 1. Architecture — what this project actually is

- **React Native app built with Expo Managed Workflow.**
- All JS/TS/TSX (screens, navigation, business logic, API calls) lives under `mobile/` and is edited in **Replit**, not Xcode.
- The `mobile/ios/` folder (where Xcode is currently open) is **generated by `expo prebuild`** and is in `.gitignore`. Anything you write directly into it will be wiped the next time prebuild runs — unless it's inside a properly registered extension target or written by a config plugin.
- Production builds are produced by **EAS Build** (Expo Application Services, cloud build). Local Xcode builds are for development and validation only.
- **Bundle ID:** `com.jobrunner.app`
- **Apple Team ID:** `6557LUX45F`
- **Current version:** 1.1.3 (build 32) — see `JobRunner/JobRunner/Info.plist`
- **JS engine:** Hermes. **New Architecture:** enabled (`RCTNewArchEnabled: true`).
- **iOS deployment target (main app):** 16.0 — set via `expo-build-properties` in `mobile/app.json` (the Podfile's `15.1` default is overridden). The Live Activity extension will need 16.1+ — small bump.
- **Liquid Glass already in use:** `@callstack/liquid-glass` is a dependency. Any new native UI should match that aesthetic (iOS 26 design language).

## 2. What Xcode is used for — and ONLY this

Xcode (and you, the Xcode Claude Agent) should be used for things Expo SDK genuinely cannot do:

- **Native extension targets** — Live Activities, WidgetKit widgets, App Clips, Share Extensions, Notification Service Extensions, etc.
- **Swift/Obj-C files that live inside one of those extension targets** (these are kept by the config plugin and survive prebuild).
- **Entitlements and capabilities** that need an extension-side counterpart (App Groups, Push Notifications, Sign in with Apple, etc.).
- **App Groups wiring** (`group.com.jobrunner.app`) for sharing data between the RN app and an extension.
- **Quick validation** via the in-Xcode build/run cycle while iterating on an extension.

## 3. What NOT to touch from Xcode

These are owned by Expo / EAS / Replit. Editing them locally either gets overwritten or breaks the build pipeline.

- ❌ `JobRunner/JobRunner/AppDelegate.swift` — managed by Expo (extends `ExpoAppDelegate`). The comment `// Extension point for config-plugins` is the official customization seam — anything you need here goes through a config plugin instead.
- ❌ `JobRunner/JobRunner/Info.plist` — regenerated from `mobile/app.json` + plugins.
- ❌ `JobRunner/JobRunner/SplashScreen.storyboard`, `Images.xcassets`, `Expo.plist`, `PrivacyInfo.xcprivacy` — all Expo-managed.
- ❌ `JobRunner/JobRunner/ExpoModulesProviders/**` — autogenerated by Expo Modules autolinking.
- ❌ Anything under `Pods/` — owned by CocoaPods + Expo autolinking.
- ❌ `Podfile` — generated by Expo's prebuild template.
- ❌ The `.xcodeproj` / `.xcworkspace` file structure for the main app target.

If something needs to change in any of the above, the change goes in **`mobile/app.json`** or a **config plugin under `mobile/plugins/`**, not in Xcode.

## 4. The only files worth committing from this work

When you make changes that need to survive prebuild and ship to EAS, only these get committed to git:

- ✅ `mobile/plugins/*.js` — config plugins (this is the bridge that tells EAS "add this extension target with these entitlements")
- ✅ `mobile/app.json` — Expo config (plugin registration, capabilities, entitlements)
- ✅ Swift source files for extension targets, **kept in a stable location the config plugin copies into the generated `ios/`** (commonly under `mobile/targets/<ExtensionName>/`)

Swift files inside the generated `ios/` folder are scratch — they exist so Xcode can build, but the source of truth for them is the location the config plugin copies from.

## 5. Active workstream — Live Activity (iOS 16.1+)

We are currently planning a **Live Activity + Dynamic Island** extension that shows an in-progress job on the lock screen and Dynamic Island.

- New target name (planned): `JobRunnerLiveActivity` (Widget Extension with ActivityKit)
- New bundle ID (planned): `com.jobrunner.app.liveactivity`
- Deployment target for the extension: **iOS 16.1+** (Live Activities are not available before that). The main app stays at iOS 15.1.
- Shared App Group (planned): `group.com.jobrunner.app`
- Will require `NSSupportsLiveActivities` (and likely `NSSupportsLiveActivitiesFrequentUpdates`) in the main app's `Info.plist`, set via the config plugin — not edited directly in `Info.plist`.
- JS side will use an Expo Modules wrapper (custom, or a community package like `expo-live-activity` / `@bacons/expo-live-activity`) to start/update/end activities from React Native.

**Update approach (v1): local-only.** The host app calls `Activity.update()` from JS via the Expo module bridge. No APNs push tokens, no server-side push work. APNs-driven updates are deferred to v2 — the extension target doesn't need to change when we add them later, only the server and the JS side. Do not wire up `pushTokenUpdates` / `PushType.token` handling in the v1 build.

**Apple Developer portal prerequisites** (must be done in this order before any EAS build of the extension will succeed):
1. Create App Group `group.com.jobrunner.app` under Identifiers → App Groups.
2. Edit existing App ID `com.jobrunner.app` → enable App Groups capability → tick `group.com.jobrunner.app`. **Easy to miss this step** when focused on creating the new identifier.
3. Create new App ID `com.jobrunner.app.liveactivity` → enable App Groups → tick `group.com.jobrunner.app`. Push Notifications capability is **not** required for the v1 local-only approach.

## 8. Quality bar

- Production-quality only — JobRunner is a paid app for Australian tradies; no placeholder UI, no `// TODO` left in shipping code.
- SwiftUI for any new view code, following the rules in the user's Xcode CLAUDE config:
  - PascalCase types, camelCase members
  - `@State private var` for SwiftUI state, `let` for constants
  - 4-space indent
  - Prefer `async`/`await` over Combine
  - Avoid force unwrapping
- Match the visual language of the main app (modern, clean, premium feel — Liquid Glass-aware where appropriate on iOS 26+).

## 9. Workflow when adding a native feature

1. **Decide if it's truly native-only.** If Expo SDK or a community Expo module can do it, that goes to Replit instead.
2. **Write the config plugin first** in `mobile/plugins/` — this is what tells EAS to create the target on cloud builds.
3. **Add the Swift extension target in Xcode** for local iteration, keeping source files under `mobile/targets/<Name>/` so the plugin can copy them in.
4. **Wire shared state via App Groups** (`group.com.jobrunner.app`), not via files in the app sandbox.
5. **Validate locally with `BuildProject`** in Xcode, then commit only the plugin + Swift sources + `app.json` changes. Never commit `ios/`.

## 10. Tools & validation preferences (for you, the Xcode Claude Agent)

- Prefer **`XcodeRefreshCodeIssuesInFile`** for fast Swift diagnostics before committing to a `BuildProject` run.
- Use **`DocumentationSearch`** liberally for ActivityKit, WidgetKit, App Intents, and anything Liquid Glass / iOS 26 — these APIs evolve fast and training-cutoff knowledge is often stale.
- Use **`ExecuteSnippet`** for quick "does this Swift expression work" checks instead of spinning up a full build.
- Reserve **`BuildProject`** for full validation — it's slow.
